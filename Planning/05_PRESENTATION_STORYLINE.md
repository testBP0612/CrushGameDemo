# 05 — 比賽簡報故事線（Presentation Storyline）

> 目的：向評審清楚展示「**可玩原型 + AI 工作流 + 治理**」三位一體，而非只 demo 一款小遊戲。

## 簡報主軸（一句話）
> 「我們做的不是一款遊戲，而是一條**能把企劃書變成可玩原型的 AI 生產線**——這款 H5 遊戲只是它的第一個產物。」

## 建議流程（約 8–10 分鐘）
1. **痛點（30s）**：AI 做遊戲常見問題——發散、改方向、context rot、難協作。
2. **我們的解法（1m）**：展示 repo 結構圖（`README.md` 導覽）。強調「文件即協作介面」。
3. **工作流走一遍（2m）**：
   - `Planning/00_GAME_PITCH` → 企劃。
   - `Docs/` 規格 + `Data/*.json` 單一真實來源。
   - `Codex/` 任務卡（展示一張卡的清楚邊界：範圍/不做/驗收）。
   - `VALIDATION_CHECKLIST` 驗收。
4. **治理亮點（2m）**：
   - `OPEN_QUESTIONS`：AI 不猜、先問。
   - `DECISIONS`：人類拍板、可追溯。
   - **Art Contract Freeze**：規格鎖定、變更要走流程——展示「防 AI 擅改」。
5. **多人多 AI 協作（1m）**：Codex 寫程式、Magnific 出美術、placeholder-first 平行不互卡。
6. **實機 Demo（2m）**：H5 開瀏覽器，玩一局風險撤離：下注→連擊→撤退 vs 繼續→結算。
7. **可複製性（1m）**：換一款遊戲只要換 `Data/`+`Docs/`+任務卡，治理骨架不變。
8. **Future（30s）**：登入、排行榜、正式美術、API——架構已預留。

## Demo 腳本（實機）
- 下注 50 → 連勝 3 關，倍率跳升 → 故意「再賭一把」 → 失敗，展示風險張力。
- 再玩一局 → 連勝後「撤退領取」 → 展示落袋為安與最佳紀錄更新。
- 重整瀏覽器 → 餘額/紀錄保留（local 服務）。

## 評審可能提問與回答備忘
- **Q：AI 會不會亂改方向？** → 展示鐵則 + OPEN_QUESTIONS + Freeze。
- **Q：美術怎麼接？** → Art Contract（規格鎖定）+ Manifest + placeholder fallback，缺圖不崩。
- **Q：能不能換別款遊戲？** → 是，工作流與遊戲解耦，換資料與任務卡即可。

## 一頁式收尾訊息
> 「企劃書 → 可玩原型，中間每一步都有文件、有邊界、有驗收、有決策紀錄。這就是可複製的 AI 遊戲生產線。」
